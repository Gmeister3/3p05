\begin{abstract}
This document describes the redesign and implementation of a networked console-based village
war strategy game for COSC 3P91 Assignment~4.  Building on Assignment~3, the application
has been split into a \textbf{client} and a \textbf{server} communicating over a \textbf{TCP}
socket connection.  All game logic (simulation, combat, time-ticking) executes on the server;
the client is a thin console front-end that sends JSON requests and displays server replies.
A handshake-based \textbf{authentication} protocol (LOGIN/REGISTER) gates access to game
commands.  The server uses two layers of \textbf{multi-threading}: a cached thread pool for
per-client connections and a fixed thread pool for CPU-intensive game operations (village
generation, army generation, attacks, and village testing).  Two new features are added:
\textbf{Generate Army} (creates a calibrated opponent army to test the player's defences) and
\textbf{Test Village} (runs multiple generated armies against the player's village and returns
a success/fail score).  All Assignment~3 design patterns (MVC, Factory, Adapter) and
Assignment~2 OOP concepts are retained.
\end{abstract}

\section{Client/Server Architecture}
% ------------------------------------------------------------------

\subsection{Design Overview}

The Assignment~4 redesign splits the single-JVM application into two separately
runnable programs.  The \textbf{server} owns all game logic, NPC simulation, and
the user database; it never renders a console menu.  The \textbf{client} handles
all user interaction (menu display, input collection) and forwards every user
command as a JSON message to the server over a TCP socket.  The server executes
the request, mutates the relevant \texttt{GameSession}, and sends back a JSON
response whose \texttt{"message"} field the client prints verbatim.

This split is coherent with the existing MVC structure: the Model and
Controller layers move entirely to the server, while the View layer
(\texttt{GraphicalInterface}) and user input remain on the client.  Network
I/O is the only new dependency.

\subsection{Socket Choice -- TCP}

TCP was chosen over UDP for the following reasons:
\begin{itemize}[noitemsep]
  \item Game commands (\texttt{BUILD}, \texttt{TRAIN}, \texttt{ATTACK}, \ldots)
        must arrive \emph{exactly once} and \emph{in the correct order}; a
        dropped or reordered packet would corrupt a player's village state.
  \item TCP's built-in flow control and retransmission eliminate the need for
        an application-layer reliability protocol, keeping the code simpler.
  \item The game is turn-based rather than real-time; TCP's slightly higher
        per-packet overhead is negligible compared with the cost of the game
        operations themselves.
\end{itemize}

The server opens a \texttt{ServerSocket} on port~9090 (configurable).  Each
\texttt{Socket} returned by \texttt{accept()} is wrapped in \texttt{BufferedReader}
and \texttt{PrintWriter} streams so messages are exchanged as newline-terminated
lines.

\subsection{Communication Protocol -- JSON Messages}

Every message exchanged between the client and the server is a single-line JSON
object serialised by \texttt{network.JsonHelper} and terminated by a newline
character.  The \texttt{network.Message} class (Table~\ref{tbl:protocol}) is an
immutable carrier built with a fluent \texttt{Builder}.  All field values are
strings, keeping the codec dependency-free.

\begin{table}[h]
  \centering
  \caption{Subset of the client/server protocol.}
  \label{tbl:protocol}
  \begin{tabularx}{\linewidth}{llX}

    \midrule
    C $\to$ S & \texttt{LOGIN}         & \texttt{username}, \texttt{password} \\

\section{Authentication}
% ------------------------------------------------------------------

Before any game command is accepted, the client must complete a
\emph{handshake-like} authentication sequence:

\begin{enumerate}[noitemsep]
  \item The client prompts the player for a username and password and sends a
        \texttt{LOGIN} (or \texttt{REGISTER}) JSON message.
  \item The server looks up the credentials in \texttt{server.PlayerDatabase}, a
        \texttt{ConcurrentHashMap<String, String>} seeded with three default
        accounts (\texttt{alice/alice123}, \texttt{bob/bob456},
        \texttt{admin/admin}).
  \item On success the server replies \texttt{LOGIN\_OK} (or
        \texttt{REGISTER\_OK}) and creates a \texttt{GameSession} for that
        connection.  On failure it replies \texttt{LOGIN\_FAIL} (or
        \texttt{REGISTER\_FAIL}) and the client prompts again.
  \item All subsequent messages from an unauthenticated connection are rejected
        with a \texttt{RESPONSE\{status:ERROR\}} reply.
\end{enumerate}

The assignment explicitly states that ``text-based comparisons suffice'' for
authentication, so no encryption is applied.

\begin{spacing}{0.8}
\begin{lstlisting}[language=Java, caption={Server authentication logic in ClientHandler.},
                   label=lst:auth]
if (session == null
        && request.getType() != MessageType.LOGIN
        && request.getType() != MessageType.REGISTER) {
    send(out, Message.error("Please authenticate first."));
    continue;
}
// handleLogin()
if (database.authenticate(username, password)) {
    session = new GameSession(username, connectionId);
    send(out, new Message.Builder(MessageType.LOGIN_OK)
            .put("message", "Welcome, " + username + "!")
            .build());
} else {
    send(out, new Message.Builder(MessageType.LOGIN_FAIL)
            .put("reason", "Invalid credentials.")
            .build());
}
\end{lstlisting}
\end{spacing}

\section{Multi-Threaded Server}
% ------------------------------------------------------------------

\subsection{Thread Model}

The server uses two layers of parallelism (Table~\ref{tbl:threads}):

\begin{table}[h]
  \centering
  \caption{Thread pools in GameServer.}
  \label{tbl:threads}
  \begin{tabularx}{\linewidth}{llX}
    \toprule
    \textbf{Pool} & \textbf{Type} & \textbf{Purpose} \\
    \midrule
    \texttt{clientPool} & \texttt{newCachedThreadPool()} &
      One thread per connected client.  Handles the JSON
      read/write loop for that connection.  Unbounded so any number
      of simultaneous clients is supported. \\
    \texttt{taskPool}   & \texttt{newFixedThreadPool(4)} &
      Shared across all clients.  CPU-intensive operations
      (village generation, army generation, attack resolution,
      village testing) are submitted as \texttt{Callable} tasks so
      that the per-client I/O thread does not block. \\
    \bottomrule
  \end{tabularx}
\end{table}

\begin{spacing}{0.8}
\begin{lstlisting}[language=Java, caption={Server accept loop and thread pool submission.},
                   label=lst:threads]
// GameServer.run() -- accept loop
while (running) {
    Socket clientSocket = serverSocket.accept();
    int id = connectionCounter.getAndIncrement();
    ClientHandler handler =
        new ClientHandler(clientSocket, database, taskPool, id);
    clientPool.submit(handler); // each client gets its own thread
}

// ClientHandler -- offload heavy attack to task pool
Future<String> future = pool.submit(new Callable<String>() {
    public String call() throws Exception {
        AttackOutcome outcome =
            session.getArmy().attack(target);
        return resultString;
    }
});
String result = future.get();
send(out, Message.ok(result));
\end{lstlisting}
\end{spacing}

\subsection{Thread Safety}

Each client has its own isolated \texttt{GameSession}; no game state is shared
between sessions, so session-level data requires no synchronisation.
\texttt{PlayerDatabase} is backed by \texttt{ConcurrentHashMap}, which makes
concurrent \texttt{authenticate()} and \texttt{register()} calls safe without
explicit locking.  The \texttt{connectionCounter} in \texttt{GameServer} uses
\texttt{AtomicInteger.getAndIncrement()} for lock-free unique ID generation.

\section{New Game Features (Assignment~4)}
% ------------------------------------------------------------------

\subsection{Generate Army}

The \texttt{GENERATE\_ARMY} command asks the server to create a detached
\texttt{Army} calibrated to the player's current village defence score, so the
player can see what kind of attacker their defences would need to repel.  The
generation is submitted to the shared \texttt{taskPool} as a \texttt{Callable}
task and the result is returned asynchronously via \texttt{Future.get()}.

\subsection{Test Village}

The \texttt{TEST\_VILLAGE} command runs a series of generated compatible armies
against the player's village and computes an overall success/fail score.  Like
attack resolution, this is executed via the shared \texttt{taskPool} so that
multiple village tests can run in parallel across different client sessions.
The server returns the number of attacks repelled and the percentage of attacks
successfully defended.

\section{Retained Assignment~3 Design Patterns}
% ------------------------------------------------------------------

\subsection{MVC Architectural Pattern}

The MVC separation introduced in Assignment~3 is maintained in the networked
design.  On the server, \texttt{ClientHandler} plays the role of the
Controller: it receives a decoded \texttt{Message}, invokes the appropriate
Model API (\texttt{Village}, \texttt{Army}, \texttt{GameEngine}), and
serialises the result.  On the client, \texttt{GameClient} plays the role of
the View: it prompts for input, sends the encoded request, and prints the
server's \texttt{"message"} field.

\subsection{Factory Creational Pattern}

\texttt{BuildingFactory} and \texttt{HabitantFactory} are used server-side
inside \texttt{ClientHandler} to instantiate buildings and habitants from the
type strings extracted from incoming JSON messages.  The handler never calls
\texttt{new} on concrete classes directly.

\subsection{Adapter Structural Pattern}

\texttt{utility.ChallengeDecisionAdapter} continues to bridge the game model
(\texttt{Army}/\texttt{Village}) to the unmodified external
\texttt{ChallengeDecision} engine.  The adapter is invoked inside the
\texttt{Callable} submitted to \texttt{taskPool} when processing
\texttt{ATTACK} requests.

\subsection{XML Persistence}

\texttt{persistence.VillageSerializer} is used server-side: the \texttt{SAVE}
command persists the session's village to
\texttt{village\_\{username\}\_save.xml} and the \texttt{LOAD} command
reconstructs it, keeping all persistent state server-side.

\subsection{Generics}

\texttt{Repository<T>} (instantiated as \texttt{Repository<Player>} and
\texttt{Repository<Village>} in \texttt{GameEngine}) and the upper-bounded
wildcard \texttt{Army.addAll(List<? extends Fighter>)} are unchanged.

\begin{spacing}{0.8}
\begin{lstlisting}[language=Java, caption={Generic Repository (excerpt).}, label=lst:repository]
public class Repository<T> {
    private final TreeMap<String, T> store = new TreeMap<>();

    public List<T> findWhere(Predicate<T> predicate) {
        return store.values().stream()
                .filter(predicate)
                .collect(Collectors.toList());
    }
}
\end{lstlisting}
\end{spacing}

\subsection{Local and Anonymous Classes}

Local classes \texttt{NpcBuilder} (\texttt{GameEngine}) and
\texttt{BuildingMerger} (\texttt{Defence}) are retained.  Anonymous
\texttt{Comparator} instances remain in \texttt{GameEngine.getAvailableTargets()}
and \texttt{Village.getBuildingsSortedByLevel()}.  In the networking layer,
anonymous \texttt{Callable<String>} instances are created inline in
\texttt{handleAttack()}, \texttt{handleGenerateArmy()}, and
\texttt{handleTestVillage()} when submitting tasks to the thread pool.

\begin{spacing}{0.8}
\begin{lstlisting}[language=Java, caption={Anonymous Callable submitted to thread pool.},
                   label=lst:callable]
Future<String> future = pool.submit(new Callable<String>() {
    public String call() throws Exception {
        AttackOutcome outcome =
            session.getArmy().attack(target);
        // build and return result string
    }
});
String result = future.get();
\end{lstlisting}
\end{spacing}

\subsection{Lambda Expressions and Method References}

Stream pipelines in \texttt{Village}, \texttt{Army}, \texttt{GameEngine},
\texttt{CollectResources}, and \texttt{Defence} are unchanged.  Highlights:

\begin{itemize}[noitemsep]
  \item \texttt{mapToDouble(b -> b.getLevel() * b.getHitPoints()).sum()} --
        \texttt{Village.getDefenceScore()}.
  \item \texttt{mapToDouble(Fighter::damage).sum()} --
        \texttt{Army.recalculateDamage()}.
  \item \texttt{sorted(Comparator.comparingInt(Player::getScore).reversed())}
        -- \texttt{GameEngine.getLeaderboard()}.
  \item \texttt{forEach(archerTowers::add)} -- \texttt{Defence.syncWithVillage()}.
\end{itemize}

\subsection{Custom Exceptions}

\texttt{InsufficientResourcesException}, \texttt{BuildingLimitExceededException},
\texttt{MaxLevelReachedException}, and \texttt{InvalidOperationException} are
still thrown by the Model.  \texttt{ClientHandler} catches them and converts
each to a \texttt{Message.error(e.getMessage())} response.

\subsection{Java Utility Classes (Collections)}

\texttt{ArrayList}, \texttt{TreeMap}, \texttt{Collections.unmodifiableList},
and \texttt{Collectors.groupingBy} are retained.
\texttt{ConcurrentHashMap} is added in \texttt{PlayerDatabase} for thread-safe
account storage.  \texttt{AtomicInteger} is used in \texttt{GameServer} for
lock-free connection-ID generation.

\subsection{I/O and Streams}

Console I/O moves to the client (\texttt{GameClient} wraps
\texttt{Scanner} over \texttt{System.in}).  Network I/O uses
\texttt{BufferedReader}/\texttt{PrintWriter} wrapping the socket streams.
The Stream API is unchanged in the Model layer.

\begin{lstlisting}[language=Java, caption={Stream-based resource collection.}, label=lst:streams]

\section{Conclusions}
% ------------------------------------------------------------------

The Assignment~4 redesign converts the single-JVM game into a fully networked
client/server application:

\begin{itemize}[noitemsep]
  \item \textbf{TCP sockets} -- reliable, ordered delivery without an
        application-layer retransmission protocol.
  \item \textbf{JSON protocol} -- dependency-free flat key-value messages
        serialised by \texttt{JsonHelper}; all 16 message types are defined in
        the \texttt{MessageType} enum.
  \item \textbf{Authentication} -- handshake-based LOGIN/REGISTER gating backed
        by \texttt{ConcurrentHashMap} in \texttt{PlayerDatabase}.
  \item \textbf{Multi-threading} -- cached client pool (one thread per
        connection) plus fixed task pool (four threads shared across sessions)
        for CPU-intensive operations; thread safety via session isolation,
        \texttt{ConcurrentHashMap}, and \texttt{AtomicInteger}.
  \item \textbf{New features} -- \texttt{GENERATE\_ARMY} and
        \texttt{TEST\_VILLAGE} extend the game with Assignment~4's required
        defence-testing functionality.
  \item All Assignment~3 design patterns (MVC, Factory, Adapter, XML
        persistence) and Assignment~2 OOP concepts are retained.
\end{itemize}

public GameServer(int port) throws IOException {
    this.serverSocket = new ServerSocket(port);
    this.database     = new PlayerDatabase();
    this.taskPool     = Executors.newFixedThreadPool(POOL_SIZE);
    this.clientPool   = Executors.newCachedThreadPool();

public void run() {
    while (running) {
        Socket clientSocket = serverSocket.accept();
        int id = connectionCounter.getAndIncrement();
        ClientHandler handler =
            new ClientHandler(clientSocket, database, taskPool, id);
        clientPool.submit(handler);
    }
}
